## 问题背景

在生产环境中，监控告警系统是保障服务稳定运行的第一道防线。然而，很多团队面临着这样的困境：

- 商业监控工具成本高，配置复杂
- 开源监控方案学习曲线陡峭
- 简单的脚本监控功能单一，无法形成体系
- 告警风暴频发，运维人员疲于奔命

本文基于我在生产环境的实战经验，从零开始搭建一套**轻量级、自动化、可维护**的监控告警系统。

## 需求分析

### 核心需求

1. **系统资源监控**
   - 内存使用率（阈值：80%）
   - CPU 负载（阈值：90%）
   - 磁盘空间（阈值：85%）

2. **服务健康检查**
   - Web 服务端口（3000, 8081, 8082）
   - 数据库连接（PostgreSQL, Redis）
   - 进程状态（PM2 管理的服务）

3. **告警通知**
   - 自动记录告警日志
   - 支持扩展邮件/短信/钉钉通知
   - 告警分级（警告、严重）

4. **故障自愈**
   - 服务宕机自动重启
   - 端口占用自动清理
   - 定时任务自动执行

### 技术选型

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| Prometheus + Grafana | 功能强大，生态完善 | 配置复杂，资源占用高 | ❌ |
| Zabbix | 企业级，功能全面 | 学习成本高，部署复杂 | ❌ |
| Shell 脚本 + Cron | 轻量级，易维护，零成本 | 功能相对简单 | ✅ |

**选择理由：**
- 初期业务简单，不需要重型监控
- 团队熟悉 Shell 脚本，维护成本低
- 零成本，适合个人项目和小团队

## 核心实现

### 1. 系统资源监控

```bash
check_memory() {
    log "📊 检查内存使用..."
    local used_percent=$(free | grep Mem | awk '{printf "%.0f", $3/$2 * 100.0}')
    
    if [ "$used_percent" -ge "$ALERT_THRESHOLD_MEMORY" ]; then
        alert "内存使用率过高：${used_percent}% (阈值：${ALERT_THRESHOLD_MEMORY}%)"
        return 1
    else
        log "✅ 内存正常：${used_percent}%"
        return 0
    fi
}

check_cpu() {
    log "📊 检查 CPU 使用..."
    local load=$(cat /proc/loadavg | awk '{print $1}')
    local cpu_count=$(nproc)
    local load_percent=$(echo "$load $cpu_count" | awk '{printf "%.0f", ($1/$2)*100}')
    
    if [ "$load_percent" -ge "$ALERT_THRESHOLD_CPU" ]; then
        alert "CPU 负载过高：${load_percent}%, 负载：$load"
        return 1
    else
        log "✅ CPU 正常：${load_percent}%"
        return 0
    fi
}

check_disk() {
    log "📊 检查磁盘使用..."
    local disk_percent=$(df -h / | awk 'NR==2 {print $5}' | sed 's/%//')
    
    if [ "$disk_percent" -ge "$ALERT_THRESHOLD_DISK" ]; then
        alert "磁盘空间不足：${disk_percent}%"
        return 1
    else
        log "✅ 磁盘正常：${disk_percent}%"
        return 0
    fi
}
```

**关键点：**
- 使用 `free` 命令获取内存信息
- 使用 `/proc/loadavg` 获取 CPU 负载
- 使用 `df` 命令获取磁盘使用率
- 所有阈值可配置，避免硬编码

### 2. 服务健康检查

```bash
check_service() {
    local name="$1"
    local port="$2"
    
    log "📊 检查服务：$name (端口 $port)"
    
    if ! nc -z localhost "$port" 2>/dev/null; then
        alert "服务宕机：$name (端口 $port)"
        
        # 尝试自动重启
        log "🔄 尝试重启服务：$name"
        pm2 restart "$name" 2>/dev/null || true
        
        sleep 5
        
        # 验证重启是否成功
        if ! nc -z localhost "$port" 2>/dev/null; then
            alert "服务重启失败：$name"
            return 1
        else
            log "✅ 服务重启成功：$name"
        fi
    else
        log "✅ 服务正常：$name"
        return 0
    fi
}
```

**关键点：**
- 使用 `nc` 命令检查端口连通性
- 服务宕机时自动尝试重启
- 重启后验证是否成功
- 记录详细日志便于排查

### 3. PM2 状态监控

```bash
check_pm2_status() {
    log "📊 检查 PM2 服务状态..."
    
    # 检查 PM2 进程重启次数
    local restart_info=$(pm2 list --no-color | grep "starlog-frontend" | awk '{print $8}')
    
    if [ -n "$restart_info" ] && [ "$restart_info" -gt 50 ]; then
        alert "PM2 服务异常重启：starlog-frontend 已重启 $restart_info 次"
        return 1
    else
        log "✅ PM2 状态正常"
        return 0
    fi
}
```

**关键点：**
- 监控 PM2 重启次数
- 重启次数过多说明有潜在问题
- 及时告警便于排查根因

### 5. 定时任务配置

```bash

# 每 5 分钟执行一次监控告警
*/5 * * * * /home/admin/.openclaw/workspace/starLog/scripts/monitor-alert.sh

# 每天凌晨 3 点执行数据库备份
0 3 * * * /home/admin/.openclaw/workspace/starLog/scripts/backup-db.sh

# 每周日凌晨 2 点清理 PM2 日志
0 2 * * 0 pm2 flush

# 每天凌晨 4 点重启前端服务（预防内存泄漏）
0 4 * * * pm2 restart starlog-frontend
```

**关键点：**
- 监控频率：5 分钟（平衡及时性和资源消耗）
- 备份时间：凌晨 3 点（业务低峰期）
- 日志清理：每周一次
- 预防性重启：每天一次

## 实战案例

### 案例 1：端口占用导致服务重启 133 次

**现象：**
- PM2 显示前端服务重启 133 次
- 错误信息：`EADDRINUSE: address already in use :::3000`

**排查：**
```bash
# 1. 查看日志
pm2 logs starlog-frontend --lines 100

# 2. 检查端口占用
netstat -tlnp | grep 3000

# 3. 发现僵尸进程
PID: 349782
进程名：next-server (v15.5.12)
启动时间：04:00 (已运行 4 小时)
```

**根因：**
- 手动启动的 Next.js 进程（PID 349782）占用端口 3000
- PM2 无法绑定端口，每 5 秒重启一次
- 循环 133 次，直至今日

**解决：**
```bash
# 终止僵尸进程
kill -9 349782

# 重启 PM2 服务
pm2 restart starlog-frontend

# 创建预防脚本
./scripts/cleanup-port.sh 3000
```

**预防：**
- 启动脚本增加端口预检查
- 自动检测并清理僵尸进程
- 禁止手动启动服务（必须通过 PM2）

### 案例 2：数据库连接失败

**现象：**
- 应用报错：`connect ECONNREFUSED 127.0.0.1:5432`
- 监控告警：PostgreSQL 容器未运行

**排查：**
```bash
# 1. 检查容器状态
docker ps | grep postgres

# 2. 查看容器日志
docker logs starlog-postgres --tail 50

# 3. 发现内存不足导致容器崩溃
```

**解决：**
```bash
# 重启容器
docker restart starlog-postgres

# 增加系统内存或优化数据库配置
```

**预防：**
- 监控脚本增加数据库容器检查
- 设置内存告警阈值
- 配置 Docker 自动重启策略

## 最佳实践

### ✅ 应该做的

1. **阈值设置合理**
   - 参考历史数据
   - 预留缓冲空间
   - 避免阈值过紧导致告警风暴

2. **告警分级**
   - 警告（Warning）：需要关注但不紧急
   - 严重（Critical）：需要立即处理
   - 灾难（Disaster）：系统不可用

3. **告警降噪**
   - 设置告警频率限制
   - 相同告警合并
   - 非工作时间降低频率

4. **故障自愈**
   - 简单故障自动恢复
   - 复杂故障人工介入
   - 记录自愈日志

5. **定期演练**
   - 模拟故障场景
   - 验证告警有效性
   - 优化响应流程

### ❌ 避免的

1. **告警疲劳**
   - 阈值过紧，告警频繁
   - 运维人员麻木，忽略真正的问题

2. **只监控不告警**
   - 记录了问题但没人知道
   - 错过最佳处理时机

3. **告警无分级**
   - 所有告警同等对待
   - 无法区分轻重缓急

4. **无故障自愈**
   - 所有问题都需要人工处理
   - 运维工作量大

5. **无文档记录**
   - 故障处理经验未沉淀
   - 同样问题重复发生
