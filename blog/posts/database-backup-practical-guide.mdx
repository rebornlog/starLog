## 备份策略设计

### 3-2-1 备份原则

| 原则 | 说明 | 实施 |
|------|------|------|
| **3** 份数据 | 生产数据 + 2 份备份 | 本地备份 + 异地备份 |
| **2** 种介质 | 不同存储介质 | 磁盘 + 云存储 |
| **1** 个异地 | 至少 1 份异地备份 | 阿里云 OSS |

### 备份类型选择

| 类型 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 全量备份 | 恢复简单，可靠性高 | 占用空间大，耗时长 | 小数据量（<10GB） |
| 增量备份 | 占用空间小，速度快 | 恢复复杂，依赖链长 | 大数据量 |
| 差异备份 | 折中方案 | 实现复杂 | 中等数据量 |

**我们的选择：全量备份**
- 数据量小（<1GB）
- 恢复简单
- 可靠性高

### 备份频率

| 数据类型 | 频率 | 保留时间 |
|----------|------|----------|
| 核心数据 | 每小时 | 7 天 |
| 业务数据 | 每天 | 30 天 |
| 日志数据 | 每周 | 90 天 |

**我们的场景：**
- 每天凌晨 3 点备份
- 保留最近 7 天
- 每周日额外保留月度备份

## 实战：PostgreSQL 自动备份

### 环境信息

```bash
# 数据库信息
数据库：starlog
用户：starlog
密码：starlog123
主机：localhost:5432

# Docker 容器
容器名：starlog-postgres
镜像：postgres:15-alpine
```

### 备份脚本

```bash
#!/bin/bash

# 数据库备份脚本
# 功能：每日自动备份 PostgreSQL，保留最近 7 天

set -e

# === 配置区域 ===
BACKUP_DIR="/home/admin/.openclaw/workspace/starLog/backups"
DB_NAME="starlog"
DB_USER="starlog"
DB_HOST="localhost"
DB_PORT="5432"
RETENTION_DAYS=7
LOG_FILE="/tmp/starlog-backup.log"

# === 函数定义 ===
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "$LOG_FILE"
}

alert() {
    local message="$1"
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] 🚨 BACKUP ALERT: $message" | tee -a "$LOG_FILE"
}

# 创建备份目录
mkdir -p "$BACKUP_DIR"

# 生成备份文件名
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/starlog_${TIMESTAMP}.sql.gz"

log "=========================================="
log "🚀 开始数据库备份"
log "=========================================="
log "📦 数据库：$DB_NAME"
log "💾 备份文件：$BACKUP_FILE"

# 检查 PostgreSQL 容器是否运行
if ! docker ps | grep -q "starlog-postgres"; then
    alert "PostgreSQL 容器未运行，备份失败"
    exit 1
fi

# 执行备份
log "📝 执行数据库导出..."
if docker exec starlog-postgres pg_dump -U "$DB_USER" -h localhost "$DB_NAME" | gzip > "$BACKUP_FILE"; then
    BACKUP_SIZE=$(du -h "$BACKUP_FILE" | cut -f1)
    log "✅ 备份成功：$BACKUP_FILE (大小：$BACKUP_SIZE)"
else
    alert "备份失败：pg_dump 执行错误"
    exit 1
fi

# 验证备份文件
if [ -f "$BACKUP_FILE" ] && [ -s "$BACKUP_FILE" ]; then
    log "✅ 备份文件验证通过"
else
    alert "备份文件为空或不存在"
    exit 1
fi

# 清理过期备份
log "🧹 清理 ${RETENTION_DAYS} 天前的旧备份..."
find "$BACKUP_DIR" -name "starlog_*.sql.gz" -type f -mtime +$RETENTION_DAYS -delete
REMAINING=$(ls -1 "$BACKUP_DIR"/starlog_*.sql.gz 2>/dev/null | wc -l)
log "✅ 保留 $REMAINING 个备份文件"

# 列出所有备份
log "📋 当前备份列表："
ls -lh "$BACKUP_DIR"/starlog_*.sql.gz 2>/dev/null | awk '{print "   " $9 " (" $5 ")"}' || log "   无备份文件"

log "=========================================="
log "✅ 数据库备份完成"
log "=========================================="

exit 0
```

### 脚本说明

**核心步骤：**
1. 检查数据库容器状态
2. 使用 `pg_dump` 导出数据库
3. gzip 压缩节省空间
4. 验证备份文件完整性
5. 清理过期备份

**关键点：**
- 使用 `set -e` 遇到错误立即退出
- 所有操作记录日志
- 备份后立即验证
- 自动清理过期备份

## 灾难恢复演练

### 恢复步骤

**场景：** 数据库误删除，需要从备份恢复

**步骤 1：找到最近的备份**
```bash
ls -lht /home/admin/.openclaw/workspace/starLog/backups/
# 找到最新的备份文件
starlog_20260325_030000.sql.gz
```

**步骤 2：解压备份文件**
```bash
cd /home/admin/.openclaw/workspace/starLog/backups
gunzip starlog_20260325_030000.sql.gz
```

**步骤 3：恢复数据**
```bash
# 方法 1：使用 docker exec
docker exec -i starlog-postgres psql -U starlog -d starlog < starlog_20260325_030000.sql

# 方法 2：使用 psql 命令
psql -h localhost -U starlog -d starlog < starlog_20260325_030000.sql
```

**步骤 4：验证恢复**
```bash
# 连接数据库
docker exec -it starlog-postgres psql -U starlog -d starlog

# 检查数据
SELECT COUNT(*) FROM users;
SELECT * FROM posts ORDER BY created_at DESC LIMIT 5;
```

### 恢复时间目标（RTO）

| 数据量 | 恢复时间 | 说明 |
|--------|----------|------|
| <1GB | <5 分钟 | 小数据量快速恢复 |
| 1-10GB | 10-30 分钟 | 中等数据量 |
| >10GB | 30-60 分钟 | 大数据量 |

**我们的目标：** RTO < 5 分钟

## 备份验证

### 自动化验证脚本

```bash
#!/bin/bash

# 备份验证脚本
# 功能：验证备份文件完整性和可恢复性

BACKUP_DIR="/home/admin/.openclaw/workspace/starLog/backups"
TEST_DB="starlog_test"

log "🔍 开始验证备份..."

# 找到最新备份
LATEST_BACKUP=$(ls -t "$BACKUP_DIR"/starlog_*.sql.gz | head -1)

if [ -z "$LATEST_BACKUP" ]; then
    log "❌ 未找到备份文件"
    exit 1
fi

log "✅ 找到备份：$LATEST_BACKUP"

# 验证文件完整性
if gzip -t "$LATEST_BACKUP" 2>/dev/null; then
    log "✅ 压缩文件完整"
else
    log "❌ 压缩文件损坏"
    exit 1
fi

# 测试恢复（在测试数据库）
log "🧪 测试恢复..."
docker exec starlog-postgres psql -U starlog -c "DROP DATABASE IF EXISTS $TEST_DB;"
docker exec starlog-postgres psql -U starlog -c "CREATE DATABASE $TEST_DB;"

gunzip -c "$LATEST_BACKUP" | docker exec -i starlog-postgres psql -U starlog -d $TEST_DB

if [ $? -eq 0 ]; then
    log "✅ 恢复测试成功"
else
    log "❌ 恢复测试失败"
    exit 1
fi

# 清理测试数据库
docker exec starlog-postgres psql -U starlog -c "DROP DATABASE IF EXISTS $TEST_DB;"

log "✅ 备份验证通过"
```

### 验证频率

- **每日：** 检查备份文件是否存在
- **每周：** 验证备份文件完整性（gzip -t）
- **每月：** 完整恢复演练

## 异地备份

### 方案选择

| 方案 | 成本 | 可靠性 | 实施难度 |
|------|------|--------|----------|
| 阿里云 OSS | 低 | 高 | 简单 |
| AWS S3 | 中 | 高 | 中等 |
| 自建服务器 | 中 | 中 | 复杂 |
| 移动硬盘 | 低 | 低 | 简单 |

**推荐：阿里云 OSS**
- 成本低（约 0.12 元/GB/月）
- 可靠性高（99.999999999%）
- 实施简单

### 实施步骤

```bash
# 1. 安装 ossutil
wget http://gosspublic.alicdn.com/ossutil/install_ossutil.sh
sudo bash install_ossutil.sh

# 2. 配置凭证
ossutil config

# 3. 创建备份桶
ossutil mb oss://starlog-backup

# 4. 上传备份
ossutil cp /home/admin/.openclaw/workspace/starLog/backups/ oss://starlog-backup/ -r

# 5. 添加到备份脚本
# 在 backup-db.sh 末尾添加：
log "☁️  上传到阿里云 OSS..."
ossutil cp "$BACKUP_FILE" oss://starlog-backup/backups/
```

## 监控告警

### 备份监控

```bash
# 添加到 monitor-alert.sh

check_backup() {
    log "📊 检查备份状态..."
    
    # 检查最新备份时间
    LATEST_BACKUP=$(ls -t /home/admin/.openclaw/workspace/starLog/backups/*.sql.gz 2>/dev/null | head -1)
    
    if [ -z "$LATEST_BACKUP" ]; then
        alert "数据库备份不存在"
        return 1
    fi
    
    # 检查备份时间（是否超过 24 小时）
    BACKUP_TIME=$(stat -c %Y "$LATEST_BACKUP")
    CURRENT_TIME=$(date +%s)
    TIME_DIFF=$(( (CURRENT_TIME - BACKUP_TIME) / 3600 ))
    
    if [ $TIME_DIFF -gt 24 ]; then
        alert "数据库备份超过 24 小时未更新（当前：${TIME_DIFF}小时）"
        return 1
    fi
    
    log "✅ 备份正常：$(basename $LATEST_BACKUP) (${TIME_DIFF}小时前)"
    return 0
}
```

### 告警场景

| 场景 | 级别 | 处理方式 |
|------|------|----------|
| 备份不存在 | 严重 | 立即处理 |
| 备份超过 24 小时 | 严重 | 立即处理 |
| 备份文件损坏 | 严重 | 立即处理 |
| 备份空间不足 | 警告 | 24 小时内处理 |

## 最佳实践

### ✅ 应该做的

1. **自动化备份**
   - 使用 Cron 定时任务
   - 避免人工操作
   - 记录详细日志

2. **多重备份**
   - 本地备份 + 异地备份
   - 不同存储介质
   - 遵循 3-2-1 原则

3. **定期演练**
   - 每月恢复演练
   - 验证备份可用性
   - 更新恢复文档

4. **监控告警**
   - 监控备份状态
   - 备份失败立即告警
   - 设置多级告警

5. **权限管理**
   - 最小权限原则
   - 生产环境隔离
   - 操作审计日志

### ❌ 避免的

1. **无备份**
   - 最危险的行为
   - 一旦出事就是灾难

2. **备份不验证**
   - 备份可能损坏
   - 定期验证可用性

3. **单一备份**
   - 本地备份不够
   - 必须有异地备份

4. **手动备份**
   - 容易忘记
   - 质量无法保证

5. **无恢复演练**
   - 不知道能否恢复
   - 恢复时间不可控
